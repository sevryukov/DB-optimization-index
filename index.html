📖 Основы

Что такое индекс?

Понимание базового принципа работы индексов в базах данных

Представьте, что вы ищете информацию в книге. Без оглавления вам придётся листать все страницы подряд. С оглавлением — вы мгновенно находите нужную тему.

Индекс в базе данных работает по такому же принципу — это отдельная структура данных, которая ускоряет поиск по таблице.

Аналогия: книга и оглавление
  • Таблица = книга с тысячами страниц
  • Индекс = оглавление в конце книги
  • Полное сканирование = листать все страницы (медленно)
  • Поиск по индексу = найти в оглавлении (быстро)

Слева: полный перебор всех строк. Справа: B-дерево позволяет найти значение за O(log n) шагов

Важно понимать: индексы ускоряют чтение, но замедляют запись. При каждой вставке, обновлении или удалении данных индекс должен быть обновлён. Это главный компромисс при использовании индексов.

🔧 Тип 1

Индекс по одному столбцу

Самый простой и распространённый тип индекса

Индекс по одному столбцу — это базовая структура, которая хранит упорядоченные значения одного столбца вместе с указателями на соответствующие строки таблицы.

Синтаксис создания
SQL -- Создание индекса по одному столбцу CREATE INDEX idx_employees_email ON employees (email); -- Индекс автоматически используется при: SELECT * FROM employees WHERE email = 'ivan@example.com';

Когда использовать:

  • Частые поиски по конкретному значению (WHERE column = value)
  • Сортировка по одному столбцу (ORDER BY column)
  • Группировка по одному столбцу (GROUP BY column)
  • Первичные ключи и уникальные поля (email, логин)

Совет: первичный ключ (PRIMARY KEY) автоматически создаёт уникальный кластеризованный индекс. Вам не нужно создавать его вручную.

🔧 Тип 2

Индекс по нескольким столбцам

Композитные индексы для сложных запросов

Композитный (составной) индекс включает значения нескольких столбцов и упорядочивает строки сначала по первому столбцу, затем по второму и т.д.

Синтаксис создания
SQL -- Композитный индекс: сначала department, потом salary CREATE INDEX idx_emp_dept_salary ON employees (department, salary); -- Этот индекс эффективен для запросов: SELECT * FROM employees WHERE department = 'IT' AND salary > 100000; -- Также работает для поиска только по department: SELECT * FROM employees WHERE department = 'IT';
Правило левого префикса

Индекс (A, B, C) может использоваться, если запрос фильтрует по:

  • A — да ✅
  • A, B — да ✅
  • A, B, C — да ✅
  • B — нет ❌ (нет левого префикса)
  • B, C — нет ❌
  • A, C — частично (только по A) ⚠️

Частая ошибка: порядок столбцов в композитном индексе критически важен! Индекс (department, salary) и (salary, department) — это разные индексы, оптимизированные для разных запросов.

🔧 Тип 3

Индекс по вычисляемому столбцу

Индексирование результатов выражений и функций

Иногда нужно искать не по исходному значению, а по результату функции или выражения. Например, поиск без учёта регистра или поиск по вычисленному полю.

Пример: поиск без учёта регистра
PostgreSQL -- Без индекса: функция UPPER() на каждой строке -- это приводит к полному сканированию! SELECT * FROM users WHERE UPPER(name) = 'ИВАНОВ'; -- Создаём индекс по выражению CREATE INDEX idx_users_name_upper ON users (UPPER(name)); -- Теперь запрос использует индекс!
Другие примеры использования
SQL -- 1. Индекс по году даты CREATE INDEX idx_orders_year ON orders (EXTRACT(year FROM order_date)); -- 2. Индекс по длине строки CREATE INDEX idx_articles_title_len ON articles (LENGTH(title)); -- 3. Индекс по JSON-полю (PostgreSQL) CREATE INDEX idx_users_data_email ON users ((data->>'email')); -- 4. Индекс по географическим координатам CREATE INDEX idx_places_location ON places USING gist (location);

Важно: в запросе необходимо использовать то же самое выражение, что и в определении индекса. Индекс по UPPER(name) не поможет для запроса WHERE name = 'Иванов' — регистры разные!

Квиз Проверка понимания
Вы создали индекс ON orders (EXTRACT(year FROM order_date)). Какой запрос будет использовать этот индекс?
A SELECT * FROM orders WHERE EXTRACT(year FROM order_date) = 2024
B SELECT * FROM orders WHERE order_date >= '2024-01-01'
C SELECT * FROM orders WHERE YEAR(order_date) = 2024
🔧 Тип 4

Индекс с условием отбора

Частичные (разреженные) индексы для подмножества данных

Частичный индекс — это индекс, который включает только строки, удовлетворяющие определённому условию. Это позволяет значительно сократить размер индекса, если вас интересует только часть данных.

Синтаксис и примеры
SQL -- Индекс только для активных пользователей CREATE INDEX idx_users_active_email ON users (email) WHERE status = 'active'; -- Индекс только для незавершённых заказов CREATE INDEX idx_orders_pending ON orders (created_at) WHERE status IN ('pending', 'processing'); -- Индекс для заказов с высокой суммой CREATE INDEX idx_orders_premium ON orders (customer_id, total) WHERE total > 10000;
Преимущества частичных индексов
  • Меньший размер — индексируются только нужные строки
  • Быстрее обновление — при изменении строки вне условия индекс не затрагивается
  • Уникальность на подмножестве — можно обеспечить уникальность email только для активных пользователей
  • Экономия места на диске — особенно при большом проценте «мусорных» данных

🔧 Тип 5

Индекс с включёнными столбцами

Покрывающие индексы для избегания обратных обращений к таблице

Обычно после поиска по индексу СУБД выполняет обратное обращение к таблице (Bookmark Lookup / Heap Fetch), чтобы получить значения столбцов, которых нет в индексе. Включённые столбцы позволяют этого избежать.

Синтаксис INCLUDE
SQL Server / PostgreSQL 11+ -- Без INCLUDE: поиск по email, но нужен ещё name и phone -- → поиск в индексе + обратное обращение к таблице CREATE INDEX idx_users_email ON users (email); -- С INCLUDE: все нужные данные уже в индексе -- → только поиск в индексе (covering index) CREATE INDEX idx_users_email_covering ON users (email) INCLUDE (name, phone); -- Запрос полностью покрывается индексом: SELECT name, phone FROM users WHERE email = 'ivan@example.com';

Слева: каждое попадание в индекс требует отдельного обращения к таблице. Справа: все данные уже в индексе.

INCLUDE vs дополнительные столбцы в ключе: столбцы в INCLUDE не участвуют в поиске и сортировке, они просто «прилагаются» к каждой записи индекса. Это экономит место, так как они не дублируются на каждом уровне B-дерева.

Квиз Проверка понимания

B Это единственный способ хранить строковые данные в индексе
C Включённые столбцы обновляются реже, чем ключевые
📊 Критерии

Критерии создания индекса

Когда создавать индекс, а когда — нет

Создание индекса — это всегда компромисс. Каждый индекс ускоряет чтение, но замедляет запись и занимает место на диске. Вот ключевые критерии для принятия решения:

1

Размер таблицы

Для таблиц до ~10 000 строк полное сканирование часто быстрее использования индекса из-за накладных расходов. Индексы имеют смысл на средних и больших таблицах.

2

Селективность запроса

Какую долю строк возвращает запрос? Если запрос выбирает более 5–10% строк таблицы, полное сканирование может быть быстрее. Индексы наиболее эффективны для выборки < 1% строк.

3

Кардинальность столбца (разброс значений)

Высокая кардинальность (много уникальных значений) — отлично для индекса (email, ID). Низкая кардинальность (мало уникальных значений) — индекс может быть бесполезен (пол, статус).

4

Соотношение чтения и записи

Частые SELECT — индекс полезен. Частые INSERT/UPDATE/DELETE — каждый индекс замедляет запись. Оцените соотношение операций в вашей нагрузке.

5

Объём и стоимость хранения индекса

Каждый индекс занимает место на диске и в оперативной памяти. Большой индекс может не поместиться в кэш, что снизит его эффективность. Оценивайте стоимость хранения.

6

Частота использования в запросах

Индекс должен использоваться часто. «Мёртвые» индексы, которые никогда не задействуются планировщиком, — пустая трата ресурсов. Мониторьте использование индексов.

7

Наличие ограничений (UNIQUE, FOREIGN KEY)

Ограничение UNIQUE автоматически создаёт индекс. FOREIGN KEY в некоторых СУБД не создаёт индекс автоматически — его стоит добавить для ускорения JOIN.

Глубже: селективность и кардинальность

Два ключевых понятия для принятия решений об индексах

Селективность (Selectivity)

Селективность — это доля уникальных значений в столбце относительно общего числа строк:

Селективность = COUNT(DISTINCT column) / COUNT(*) Пример: таблица users с 1 000 000 строк — email: 1 000 000 уникальных → селективность = 1.0 (идеально) — department: 5 уникальных → селективность = 0.000005 (плохо) — gender: 2 уникальных → селективность = 0.000002 (очень плохо)

Правило: чем ближе селективность к 1.0, тем полезнее индекс по этому столбцу.

Селективность запроса

Также важна селективность самого запроса — сколько строк он возвращает:

Таблица orders: 1 000 000 строк -- Высокая селективность запроса (хорошо для индекса) SELECT * FROM orders WHERE id = 42; -- Возвращает 1 строку из 1 000 000 → 0.0001% -- Низкая селективность запроса (плохо для индекса) SELECT * FROM orders WHERE status = 'completed'; -- Возвращает 900 000 строк → 90% -- Полное сканирование будет быстрее!
Интерактивный калькулятор селективности
Селективность столбца: 50.00%
Селективность запроса: 0.01%
Рекомендация: ✅ Индекс полезен — высокая селективность столбца и малая выборка

Нюанс: даже столбец с низкой кардинальностью (например, status) может быть полезен в композитном индексе, если он стоит первым и значительно сокращает область поиска для последующих столбцов.

Стоимость индексов

Обратная сторона медали: что вы платите за ускорение чтения

Каждый индекс — это не бесплатное ускорение. Вот все виды «платы», которые вы вносите за каждый дополнительный индекс:

Стоимость записи

При каждой операции INSERT, UPDATE, DELETE все индексы таблицы должны быть обновлены:

Таблица с 5 индексами: INSERT → 1 запись в таблицу + 5 записей в индексы = 6 операций UPDATE → 1 обновление таблицы + до 5 обновлений индексов DELETE → 1 удаление из таблицы + 5 удалений из индексов ⚡ Вывод: 5 индексов = запись в ~5 раз медленнее!
Стоимость хранения

Индексы занимают место на диске и в оперативной памяти:

  • На диске: индекс обычно составляет 10–30% от размера таблицы (зависит от типа)
  • В памяти: индексы конкурируют за буферный кэш СУБД
  • Резервные копии: увеличивают время и объём бэкапов
Стоимость обслуживания
  • Дефрагментация: со временем индексы фрагментируются и требуют перестройки (REINDEX / ALTER INDEX REBUILD)
  • Статистика: СУБД должна поддерживать актуальную статистику по индексам (ANALYZE)
  • Планировщик: чем больше индексов, тем больше вариантов планирует оптимизатор запросов

Золотое правило: не создавайте индекс «на всякий случай». Каждый индекс должен быть обоснован конкретным запросом или ограничением. «Мёртвые» индексы — самая распространённая ошибка.

Как найти неиспользуемые индексы
PostgreSQL -- Показывает индексы, которые никогда не использовались SELECT schemaname, relname, indexrelname, idx_scan AS reads, pg_size_pretty(pg_relation_size(indexrelid)) AS size FROM pg_stat_user_indexes WHERE idx_scan = 0 ORDER BY pg_relation_size(indexrelid) DESC;

5 типов индексов — краткий обзор
Тип Когда использовать Пример
По одному столбцу Частые поиски по одному полю ON users (email)
По нескольким столбцам Запросы с несколькими условиями ON orders (status, date)
Вычисляемый Поиск по функции/выражению ON users (LOWER(email))
С условием Интересует только часть данных ON orders (date) WHERE status='pending'
С INCLUDE Нужны дополнительные столбцы без обращения к таблице ON users (email) INCLUDE (name, phone)
Чек-лист: стоит ли создавать индекс?
  • Таблица содержит достаточно строк (обычно > 50 000)
  • Запрос выбирает малую долю строк (< 5–10%)
  • Столбец имеет высокую кардинальность (много уникальных значений)
  • Запрос выполняется часто, а запись — относительно редко
  • Индекс поместится в оперативную память (буферный кэш)
  • Нет дублирующего индекса с тем же или более широким охватом
Лучшие практики
  • Мониторьте использование индексов и удаляйте неиспользуемые
  • Начинайте с EXPLAIN — анализируйте план выполнения запроса перед созданием индекса
  • Порядок столбцов важен — ставьте более селективные столбцы первыми в композитном индексе
  • Частичные индексы — мощный инструмент для экономии места при работе с подмножествами данных
  • INCLUDE — используйте для избежания обратных обращений к таблице в частых запросах
  • Не переиндексируйте — каждый лишний индекс замедляет запись
